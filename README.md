## 一、项目简介

这是一个**网页端的工单数据分析与复盘看板**。它的作用是：你把生产过程中产生的工单数据（Excel表格）上传到系统，系统自动帮你分析：

- 哪些类型的问题最多？
- 哪些工单"需求描述不清楚"？
- 哪些工单可能被反复打回修改？
- 哪些工单风险最高、需要优先处理？
- 整个团队的工单处理效率如何？

项目当前版本是 **V2.0+**，支持两种视图模式：
- **复盘看板**（传统模式）：适合集中复盘会议，展示全局数据
- **生产指挥舱**（新模式）：适合日常管理，实时监控工单状态

## 二、核心概念：什么是"工单"

如果你是第一次接触这个系统，先理解这几个概念：

### 2.1 工单（Work Order）

**工单**就是一条任务记录。在课程生产场景中，当有人发现课程内容有问题（比如题目文字写错了、图片显示不对、视频音效有问题），就会创建一条工单，指派给对应的人去修改。

一条工单通常包含这些信息：

| 字段 | 含义 | 示例 |
|------|------|------|
| 课程定位 | 这门课是什么产品 | `三年级系统课` |
| 年级 | 面向哪个年级 | `三年级` |
| 周 | 第几教学周 | `W1` |
| 所属类型 | 工单涉及的内容类型 | `题干/题目`、`图片/UI`、`视频/音频` |
| 问题描述 | 具体描述出了什么问题 | `第3讲第5题选项B的标点符号错误` |
| 状态 | 工单当前处理到哪一步 | `处理中`、`待教研验收`、`完成归档` |
| 负责人 | 谁负责处理这条工单 | `张三` |
| 上报人 | 谁提交的这条工单 | `李四` |

### 2.2 工单生命周期

一条工单从创建到关闭，通常经过以下阶段：

```
提交(上报) → 处理中 → 待验收 → 验收通过 → 归档(关闭)
```

### 2.3 关键术语

| 术语 | 含义 |
|------|------|
| **有效工单** | 经过系统分析后，字段完整、有实际问题的工单，计入统计数据 |
| **无效工单** | 缺少关键字段、只有占位文字、或是测试数据的记录，不参与核心统计 |
| **需求不明确** | 问题描述不够具体，处理人看了不知道该改什么 |
| **高频出错内容** | 同一类问题反复出现，说明某个环节存在系统性缺陷 |
| **反复调整候选** | 有迹象表明这个工单可能被来回修改了多次 |
| **风险等级** | 系统综合判断这条工单的紧急/重要程度，分高/中/低三档 |
| **归档** | 工单处理完成并验收通过后的最终状态 |

## 三、这个看板能做什么

一句话总结：**上传 Excel → 自动分析 → 图表展示 → 辅助决策**。

具体功能清单：

1. **上传 Excel 文件**（`.xlsx` / `.xls`），自动解析其中的工单数据
2. **自动给每条工单做"问题分类"**（比如判断是"图片问题"还是"文案问题"）
3. **自动提取问题关键词**（比如发现描述里提到了"标点"、"对齐"）
4. **自动识别"需求不明确"的工单**（描述太模糊，处理人没法下手）
5. **自动评估每条工单的风险等级**（高/中/低）
6. **自动识别"反复调整"的工单**（可能被来回打回了好几次）
7. **自动生成处理建议**（针对每类问题的改进方向）
8. **用图表展示整体数据**（饼图、柱状图、趋势图等）
9. **支持多维度筛选**（按年级、周次、类型、风险等级等筛选）
10. **支持自定义分类规则**（你可以自己定义什么叫"图片问题"）
11. **本地保存数据**（上传的数据存在本地文件中，刷新不丢失）
12. **导出复盘报告**（一键下载 HTML 格式的复盘报告）
13. **预留飞书同步功能**（后续可以直连飞书多维表格）

## 四、运行环境准备（小白必读）

### 4.1 你需要安装什么

这个项目需要你电脑上有 **Node.js** 环境。Node.js 可以理解为"让电脑能运行 JavaScript 代码的工具"。

### 4.2 安装 Node.js（Windows）

**第一步：下载安装包**

1. 打开浏览器，访问：`https://nodejs.org`
2. 推荐下载 **LTS 版本**（长期支持版，更稳定），版本号 ≥ 18
3. 点击下载 `.msi` 安装包（64-bit）

**第二步：安装**

1. 双击下载的 `.msi` 文件
2. 一路点"Next"（下一步），保持默认选项即可
3. 安装最后一步，点击"Finish"

**第三步：验证安装是否成功**

打开"命令提示符"（怎么打开：按键盘 `Win + R`，输入 `cmd`，回车），然后分别输入以下两条命令：

```bash
node -v
```

如果看到类似 `v18.17.0` 这样的版本号，说明 Node.js 安装成功。

```bash
npm -v
```

如果看到类似 `9.6.0` 这样的版本号，说明 npm（Node.js 的包管理器）也安装成功了。

> **小白提示**：`npm` 是 Node.js 自带的工具，用来下载项目需要的各种"依赖包"（别人写好的代码库）。你不需要深入了解它，知道怎么用命令就行。

### 4.3 什么是"终端"/"命令行"

本文档里所有以 `$` 或代码块形式出现的命令，都需要在"终端"里输入。在 Windows 上：

- **推荐**：在项目文件夹的地址栏输入 `cmd` 然后回车，即可打开终端（当前目录就是项目目录）
- 或者：按 `Win + R` → 输入 `cmd` → 回车 → 然后用 `cd 目录路径` 命令切换到项目目录

**重要：所有命令输入完后按回车键才会执行。**

## 五、如何启动项目

### 方式一：一键启动（推荐）

**首次使用**需要先安装依赖（就像新手机第一次要装 App）：

```bash
cd D:\A工作\复盘看板\workorder-dashboard
npm install
npm run install:all
```

> 说明：`npm install` 安装根目录需要的包，`npm run install:all` 会分别进入 `backend` 和 `frontend` 目录安装各自需要的包。整个过程可能需要 2-5 分钟（取决于网络速度）。

**之后每次启动**，只需要在项目根目录执行一条命令：

```bash
npm run dev
```

这条命令会**同时启动前端和后端**，终端里会用不同颜色区分输出：
- 🔵 **蓝色** → 后端服务 (运行在 `http://localhost:3001`)
- 🟢 **绿色** → 前端服务 (运行在 `http://localhost:5173`)

启动成功后，打开浏览器访问：**`http://localhost:5173/`**

> **小白提示**：`localhost` 就是你自己的电脑，`5173` 是端口号（可以理解为"门牌号"）。浏览器访问这个地址就是在访问你自己电脑上运行的服务。

### 方式二：分别启动（用于排查问题）

如果一键启动遇到问题，可以分别启动前端和后端来排查。

**启动后端**（先开一个新终端）：

```bash
cd D:\A工作\复盘看板\workorder-dashboard\backend
npm install
npm run dev
```

看到 `Workorder dashboard backend is running on http://localhost:3001` 说明后端启动成功。

验证后端是否正常：

浏览器访问 `http://localhost:3001/api/health`，如果看到：

```json
{ "ok": true }
```

说明后端一切正常。

**启动前端**（再开一个新终端）：

```bash
cd D:\A工作\复盘看板\workorder-dashboard\frontend
npm install
npm run dev
```

终端会输出类似 `http://localhost:5173/` 的地址，用浏览器打开即可。

> **端口冲突怎么办**：如果提示 `5173` 端口被占用，Vite 会自动换到 `5174` 或别的端口，以终端实际输出为准。

### 停止项目

在终端里按 `Ctrl + C`（同时按键盘上的 Ctrl 键和 C 键），即可停止服务。

## 六、操作指南：从上传到查看

### 第一步：启动项目

按照[第五章](#五如何启动项目)的方式启动项目。看到终端有蓝色和绿色输出后，打开浏览器访问 `http://localhost:5173/`。

### 第二步：确认后端连接

页面打开后，看左上角的状态标签：
- ✅ **"后端已连接"**（绿色）→ 一切正常
- ❌ **"后端未连接"**（红色）→ 后端没启动或端口不对，回头看第五章

### 第三步：上传 Excel 文件

1. 点击页面上方的 **"展开 数据源配置与上传"** 按钮
2. 在展开的区域中找到 **"选择 Excel 文件"** 按钮
3. 选择你的 Excel 文件（必须包含名为「工单任务」的 sheet）

> **测试用**：项目自带了一个示例文件，在 `samples/26 年暑假系统课生产.xlsx`，可以先用这个试试。

4. 点击 **"上传并分析"** 按钮
5. 等待 1-3 秒，页面会自动刷新，展示分析结果

### 第四步：查看分析结果

上传成功后，页面从上到下会依次展示：

1. **核心指标卡片** — 总工单数、归档率、验收通过率、需求不明确数量、高风险/反复调整数量
2. **左右分栏** — 左侧是结论卡片（自动轮播），右侧是工单明细表
3. **重点工单列表** — 自动滚动展示需要优先关注的工单
4. **辅助图表**（需要展开）— 饼图、柱状图、趋势图

### 第五步：筛选和深入查看

1. **点击指标卡片**：可以直接筛选对应类型的工单（比如点击"需求不明确"卡片，右侧表格只显示需求不明确的工单）
2. **点击左侧结论卡片**：会弹窗展示该类问题的详细列表
3. **使用右侧表格上方的筛选栏**：可以按年级、周次、类型、风险等级、负责人等多维度筛选
4. **点击任意工单行**：弹出详情弹窗，展示完整字段、风险原因和处理建议
5. **右上角可以切换视图**：复盘看板 vs 生产指挥舱

### 第六步：导出复盘报告

点击右上角的 **"📄 导出复盘报告"** 按钮，系统会生成一份 HTML 格式的复盘报告并自动下载。

## 七、页面功能详解

### 7.1 顶部导航栏

页面最上方（header）包含以下信息：
- **标题**：课程生产工单复盘看板
- **副标题**：看板的功能定位说明
- **后端连接状态**：绿色=已连接，红色=未连接
- **导出复盘报告按钮**：点击下载 HTML 报告
- **分类规则按钮**：点击打开分类规则管理面板
- **视图切换按钮**：在"复盘看板"和"生产指挥舱"之间切换

### 7.2 数据源与上传区域

这是一个可折叠的区域（默认折叠），包含：

- **数据源选择器**：选择要查看的数据源（当前仅支持 Excel 上传和飞书同步两种来源）
- **同步设置**：是否开启自动同步（针对飞书数据源，每 5 分钟自动拉取一次）
- **上传区域**：选择并上传 Excel 文件

> **注意**：上传新文件会**覆盖**当前数据，旧数据会被清空。

### 7.3 核心指标卡片

上传数据后，页面中间会显示 **5 个核心指标卡片**，这是整个看板最重要的数据摘要：

| 卡片 | 显示内容 | 可点击 |
|------|---------|--------|
| 总工单 | 总记录数 + 有效分析数 | ✅ 点击筛选有效工单 |
| 归档率 | 归档百分比 + 已归档/未完成数 | ❌ |
| 验收通过率 | 验收通过百分比 + 验收总数 | ❌ |
| 需求不明确 | 不明确工单数 + 占比 | ✅ 点击筛选不明确工单 |
| 高风险/反复调整 | 高风险数 / 反复调整数 + 占比 | ✅ 点击筛选高风险工单 |

### 7.4 左右分栏（结论 + 数据）

这是看板的主要展示区域，左右两栏布局：

#### 左侧：结论面板（LeftConclusionPanel）

从左到右展示三个层次的信息：

**① 状态分布条**

一条彩色的横向条，用颜色表示工单在不同状态下的分布比例：
- 🟢 **已归档**：已完成并关闭的工单
- 🟡 **待验收**：等待教研验收的工单
- 🔵 **处理中**：正在处理中的工单
- ⚪ **暂停/挂起**：暂停或被挂起的工单

点击任意颜色段可以筛选对应状态的工单。

**② 高频出错内容 Top5（自动轮播卡片）**

按"所属类型 / 问题一级分类 / 问题关键词"三个维度聚合，自动轮播展示最频繁出现的问题类型。每张卡片显示：
- 排名（TOP 1 ~ TOP 5）
- 聚合名称（如 `题干/题目 / 题干题目文案 / 标点`）
- 数量 + 占比
- 涉及的年级和周次
- 一条代表性工单的问题描述

> **这是复盘时最需要关注的部分** — 它直接告诉你"什么问题出得最多"。

**③ 需求不明确与反复调整**

用轮播卡片展示：
- 需求不明确的原因排行（如"范围过大"、"定位不具体"）
- 反复调整的工单分布

**④ 反复修改根因分布**

分析反复修改工单的可能原因：
- `unclear_requirement`（需求描述不清晰 + 反复调整）：需求本身没说清楚
- `execution_gap`（反复调整但需求已明确）：可能是执行层面有问题
- `batch_issue`（命中多个关键词 + 反复调整）：可能是批量性问题

**⑤ 改进建议**

系统根据当前数据自动生成的改进建议，比如：
- "针对「某问题」建立专项检查清单"
- "需求不明确占比过高，建议在提单阶段增加要求"
- "存在 N 条反复调整工单，建议复盘需求表达或验收口径"

#### 右侧：数据面板（RightDataPanel）

包含 **4 个 Tab 页签**：

- **全量明细**：展示所有工单的表格（带筛选器），是默认打开的视图
- **待验收队列**：只显示状态为"待教研验收"的工单，按等待天数排序
- **反复修改**：只显示被标记为"反复调整候选"的工单
- **不合格工单**：只显示被标记为"无效"的工单，按无效类型分组

全量明细表格包含以下列（可通过筛选器缩小范围）：

| 列名 | 说明 |
|------|------|
| 是否有效分析 | ✅有效 / ❌无效 |
| 风险等级 | 高🟥 / 中🟨 / 低⬜ |
| 状态 | 工单当前处理状态 |
| 状态分组 | 归档/待验收/处理中/暂停挂起 |
| 课程定位 | 课程产品名称 |
| 年级 | 面向的年级 |
| 周 | 教学周次 |
| 所属类型 | 工单涉及的内容类型 |
| 问题一级分类 | 系统自动判断的分类（鼠标悬停可看到命中的关键词） |
| 问题关键词 | 从描述中提取的关键词 |
| 是否需求不明确 | 是/否 |

## 九、工单分类逻辑详解（核心）

这一章是本文档最重要的部分，详细解释系统是如何给工单做分析的。如果你是第一次使用这个系统，建议仔细阅读。

### 9.1 问题一级分类规则

系统通过**关键词匹配**来判断一条工单属于哪类问题。当前内置了 **7 大分类**：

| 分类名称 | 匹配关键词 | 覆盖范围 |
|---------|-----------|---------|
| **负反馈/弹窗/解析** | 负反馈、弹窗、解析、错三次、错误音效 | 负反馈机制、弹窗交互、题目解析、错题反馈、错误音效 |
| **题干/题目/文案** | 题干、题目、试题、选项、文字、标点、问号、句号 | 题干内容、题目表述、试题选项、文字错误、标点符号 |
| **图片/UI显示** | 图片、截图、配图、显示、图标、小手、按钮、对齐、位置 | 页面视觉元素：图片、截图、图标、按钮样式、文字对齐、位置偏移（不含弹窗逻辑） |
| **流程/交互/阻断** | 点击、引导、关闭、跳转、进入、阻断、流程 | 用户操作流程、交互引导、页面跳转、任务阻断 |
| **音视频/语音** | 讲解视频、视频、音频、音效、语音、旁白 | 多媒体内容：讲解视频、音频、音效、语音、旁白 |
| **文件/配置/Key** | key、Key、KEY、Figma、figma、打包、课包、新建、替换、配置 | 文件配置、Key值、Figma、打包、课包、资源替换 |
| **其他** | — | 没有命中以上任何分类的工单归入此类 |

> **重要提示**：如果"其他"分类的工单占比超过 25%，系统会在左侧面板显示黄色警告："有 X 条有效工单被归为'其他'，建议优化分类规则"。这说明当前的内置关键词不够覆盖你的实际场景，需要自定义分类规则。

### 9.2 分类匹配算法

系统处理每条工单时的分类逻辑：

```
1. 取工单的「问题描述」字段全文
2. 遍历 7 大分类（按代码中定义的顺序）
3. 对每个分类的每个关键词，检查是否出现在描述中（不区分大小写）
4. 一旦某个分类有至少 1 个关键词命中 → 该工单即归入此分类
5. 所有命中的关键词都会被记录下来（作为"问题关键词"字段）
6. 如果 7 个分类都没有命中 → 归入"其他"
```

**关键细节**：
- 匹配是**不区分大小写**的：`Key`、`key`、`KEY` 都能命中
- 匹配使用**子串匹配**（`includes`）：描述中有"弹窗"就命中，不需要单词完全等于"弹窗"
- 命中的关键词会被**全量记录**：比如描述中同时出现"标点"和"文字"，两个关键词都会被记下来
- 分类**按规则顺序优先**：如果一条工单同时命中两个分类的关键词，取代码中先定义的那个分类。不过实际中各分类关键词互斥性较强，这种情况较少

### 9.3 需求不明确识别规则

系统定义了 **5 种需求不明确的情形**，每条工单可能同时命中多种：

| 不明确原因 | 匹配关键词 | 含义解释 | 示例 |
|-----------|-----------|---------|------|
| **范围过大** | 所有、全部、其余、整个、这一讲、所有讲次 | 问题描述涉及的范围太广，没有精确定位 | "这一讲的题目都有问题"（到底哪道题？） |
| **参考旧版** | 参考、旧版、原版、春季、去年、25年、一致 | 要求参考旧版本但没有附上旧版链接或截图 | "参考春季的做法改一下"（春季的放在哪？） |
| **单条多需求** | 1、/ 2、/ 3、/ ；/ 并且/ 同时/ 另外/ 还有 | 一条工单里包含了多个不同的问题 | "1、第3题文字不对；2、第5题图片位置偏了；3、第7题音效缺失" |
| **步骤依赖多** | 先、然后、之后、最后、再 | 处理步骤有先后依赖关系，容易遗漏 | "先把所有题目文字改成宋体，然后检查图片对齐，最后导出打包" |
| **定位不具体** | 不对、有问题、不清楚、不显示、不出现、异常 | 只说"有问题"但没说具体是什么问题 | "第2讲显示不对"（哪里不对？怎么不对？） |

**判断逻辑**：
- 每条工单的「问题描述」逐一与以上 5 条规则进行关键词匹配
- 只要命中 ≥1 条规则 → 标记为"需求不明确"
- 所有命中的原因都会记录下来（如 `["范围过大", "定位不具体"]`）

### 9.4 风险等级判断规则

系统对每条**有效工单**进行风险评估，分为 **高/中/低** 三个等级：

#### 🔴 高风险（满足任一条件即触发）

| 条件 | 含义 |
|------|------|
| 问题描述为空 | 没有任何问题描述 |
| 命中 ≥2 个需求不明确原因 | 描述非常模糊，既范围大又定位不清 |
| 同类问题出现 ≥3 次 **且** 状态未完成 | 同一个问题反复出现且还没修好 |
| 描述包含明显返工词（还是、仍然、重新、再改、没改到、返工、反复） | 描述中就透露出"这是第N次修改" |

#### 🟡 中风险（不满足高风险条件，但满足任一）

| 条件 | 含义 |
|------|------|
| 命中 1 个需求不明确原因 | 描述有模糊之处但不算特别严重 |
| 状态未完成 **且** 问题已归类到具体分类（非"其他"） | 有明确问题类型但还没处理完 |
| 同类问题出现 2 次 | 开始出现重复苗头 |

#### ⬜ 低风险（以上都不满足）

默认等级，未命中任何明显风险规则。

> **注意**：无效工单全部标为低风险，且不参与核心风险分析。

### 9.5 反复调整候选识别规则

系统通过**三条规则**判断一条工单是否"可能被反复调整过"：

| 检测条件 | 判定逻辑 | 示例 |
|----------|---------|------|
| **同类问题重复 ≥3 次** | 按"所属类型/问题一级分类/关键词"聚合，相同组合出现 3 次及以上 | 同类型的"题干/文案/标点"问题出现了 5 次 |
| **描述含返工信号词** | 命中：还是、仍然、重新、再改、又、没改到、继续改、二次修改、返工、反复 | "图片还是不对，没改到第5页" |
| **需求不明确原因 ≥2 个 且 状态未完成** | 需求表达不清 + 还没处理完 = 可能需要反复沟通 | 既"范围过大"又"定位不具体"且还没归档 |

**满足任意一条 → 标记为"反复调整候选"**。所有这些条件都会被记录在 `repeatedAdjustmentReasons` 字段中。

### 9.6 无效工单判定规则

系统会标记出 **不应参与核心分析** 的工单记录。判定逻辑分两步：

**第一步：检查是否命中无效原因**

| 无效原因 | 判定规则 |
|---------|---------|
| 核心字段缺失 | 课程定位、所属类型、问题描述、状态任一项为空 |
| 占位或非问题描述 | 问题描述等于：组课文档、测试、占位、无、暂无、待补充、-、/ |
| 描述过短且无法分类 | 描述 < 6 个字 且 没有被任何分类规则命中 且 不是已知的简短有效描述 |
| 无法归类且缺少关键词 | 问题分类为"其他" 且 没有提取到任何关键词 且 没有命中任何不明确原因 |

**第二步：判定无效类型**

| 无效类型 | 含义 | 匹配条件 |
|---------|------|---------|
| `collaboration_placeholder` | 协作占位 | 描述包含"组课文档"、"课程文档"、"教研文档" |
| `test_data` | 测试数据 | 描述包含"测试"、"test"、"调试"、"demo" |
| `blank` | 空白/无效 | 描述为"无"、"暂无"、"-"、"/"、"待补充"、"占位" |
| `incomplete` | 信息不完整 | 核心字段有缺失 |

满足以上条件的工单 → `isValidForAnalysis = false`，不进入核心统计。

> 页面默认只展示有效工单，你可以通过右上方的筛选器切换为"全部"或"仅无效"来查看无效工单。

### 9.7 状态分组规则

系统将工单的原始状态映射为 **4 个状态分组**：

| 状态分组 | 包含的原始状态 |
|---------|--------------|
| **已归档** | `完成归档` |
| **待验收** | `待教研验收`、`教研验收中` |
| **处理中** | `处理中`、`待更新`、`待测试`、`待处理` |
| **暂停/挂起** | `暂停`、`打包机挂起`、`下期迭代` |

如果某条工单的状态不在以上任何一组中，会被归入"其他"分组。

### 9.8 处理建议生成规则

系统根据工单的具体情况，自动生成针对性的处理建议：

| 触发条件 | 生成建议 |
|---------|---------|
| 问题描述为空 | "建议补充具体问题描述、题目位置、复现步骤和预期结果。" |
| 需求不明确原因包含"范围过大" | "建议明确具体讲次、题目、页面或修改范围。" |
| 需求不明确原因包含"参考旧版" | "建议补充旧版链接、截图或差异说明。" |
| 需求不明确原因包含"单条多需求" | "建议拆分为多个独立工单，降低遗漏和返工风险。" |
| 需求不明确原因包含"步骤依赖多" | "建议按处理顺序拆解步骤，并明确每一步完成标准。" |
| 需求不明确原因包含"定位不具体" | "建议补充年级、周次、题目位置、截图或客户端路径。" |
| 被标记为反复调整候选 | "建议优先确认需求口径、验收标准和最终修改范围。" |
| 高风险 + 状态未完成 | "建议优先确认负责人、完成时间和当前阻塞点。" |

### 9.9 返工根因推断规则

如果一条工单被标记为"反复调整候选"，系统会进一步推断反复调整的根本原因：

| 根因 | 推断条件 | 含义 |
|------|---------|------|
| `unclear_requirement` | 需求不明确 + 反复调整候选 | 需求本身没写清楚，导致沟通反复 |
| `execution_gap` | 反复调整候选 + 但需求已明确 | 需求清楚但还是反复改，可能是执行问题 |
| `batch_issue` | 命中 ≥2 个关键词 + 反复调整候选 | 可能是批量性的系统问题 |

## 十、所有统计指标说明

以下是系统计算的所有统计指标及其含义：

### 10.1 数量类指标

| 指标 | 计算公式 | 含义 |
|------|---------|------|
| `totalRawCount` | Excel 中解析出的总行数 | 上传的 Excel 一共有多少条记录 |
| `validAnalysisCount` | 总行数 - 无效工单数 | 参与核心分析的有效工单数 |
| `invalidAnalysisCount` | 总行数 - 有效工单数 | 被判定为无效的记录数 |
| `archivedCount` | 有效工单中状态为"完成归档"的数量 | 已彻底关闭的工单数 |
| `unfinishedCount` | 有效工单数 - 已归档数 | 尚未完成归档的工单数 |
| `unclearCount` | 有效工单中 `isUnclearRequirement = true` 的数量 | 需求描述不明确的工单数 |
| `highRiskCount` | 有效工单中风险等级为"高"的数量 | 需要优先关注的高风险工单 |
| `repeatedCandidateCount` | 有效工单中被标记为反复调整候选的数量 | 可能被反复修改的工单数 |

### 10.2 比率类指标

| 指标 | 计算公式 | 含义 |
|------|---------|------|
| `archiveRate` | 已归档数 ÷ 有效工单数 × 100% | 工单归档完成率，越高越好 |
| `unclearRate` | 需求不明确数 ÷ 有效工单数 × 100% | 需求不明确占比，越低越好（≥20% 警告） |
| `repeatedAdjustmentRate` | 反复调整候选数 ÷ 有效工单数 × 100% | 反复调整占比，越低越好 |
| `passRate` | 已归档数 ÷ 进入过验收的工单数 × 100% | 验收通过率（需要时间数据才能计算） |

### 10.3 排行类指标

| 指标 | 含义 |
|------|------|
| `errorContentRanking` | 按"所属类型/分类/关键词"聚合的高频问题排行（Top10） |
| `issueCategoryRanking` | 7 大问题分类的工单数量排行 |
| `typeRanking` | 所属类型的工单数量排行 |
| `gradeRanking` | 各年级工单数量排行 |
| `weekRanking` | 各教学周工单数量排行 |
| `statusRanking` | 各原始状态的工单数量排行 |
| `statusGroupRanking` | 4 大状态分组的工单数量排行 |
| `unclearReasonRanking` | 5 种需求不明确原因的命中排行 |
| `repeatedAdjustmentRanking` | 反复调整原因的排行 |
| `invalidReasonRanking` | 无效原因的排行 |
| `pendingDurationRanking` | 积压时长最久的未完成工单 Top10 |

### 10.4 时间类指标

| 指标 | 含义 |
|------|------|
| `avg` (durationStats) | 工单从提交到归档/解决的平均耗时（可读格式） |
| `median` | 中位数耗时（排除极值干扰，更稳健） |
| `min` / `max` | 最短和最长耗时（反映极端情况） |
| `p75` / `p90` | 75/90 分位数耗时（如 p90=72h 表示 90% 的工单在 72 小时内处理完） |

## 十一、图表如何解读

### 11.1 问题一级分类占比（环形图）

**怎么看**：看哪个环的占比最大，就说明哪类问题最多。

**读图示例**：如果"题干/题目/文案"占 35%，说明超过 1/3 的工单都是文字和题目相关的，可能需要加强文案审核环节。

### 11.2 所属类型分布（玫瑰图/南丁格尔图）

**怎么看**：花瓣越大 = 该类工单越多。直观展示不同类型工单的分布。

### 11.3 高频出错内容 Top10（横向柱状图）

**怎么看**：这是最直观的图表。柱子越长 = 该类问题出现次数越多。

**解读重点**：
- 如果 Top1 占比超过 20%，说明有一个非常集中的问题源
- 如果 Top10 分布均匀（都在 5%-8%），说明问题比较分散
- 点击柱子进入详情可看到该问题的代表案例

### 11.4 年级问题分布（柱状图）

**怎么看**：对比各年级的工单量。如果某个年级明显高于其他，说明该年级课程的质量问题较多。

### 11.5 周次问题趋势（折线面积图）

**怎么看**：看趋势走向。
- 📈 **上升趋势**：问题在增多，可能是质量在下降或测试覆盖面在扩大
- 📉 **下降趋势**：问题在减少，可能是质量在改善
- 🔺 **尖峰**：某周问题暴增，可能需要重点复盘那周的情况

### 11.6 工单趋势图（时间分析区域，多线折线图）

四条线：
- 🔵 总工单数 — 整体工作量
- 🟢 已归档 — 完成的数量
- 🟠 需求不明确 — 模糊需求数量
- 🔴 高风险 — 高风险工单数量

**理想模式**：总工单线趋于平稳或下降 + 已归档线上升 + 需求不明确线保持低位

### 11.7 状态分布条（左侧面板）

这是一个横向百分比条，四种颜色代表四种状态分组。如果"已归档"（绿色）占比大说明进度良好，如果"暂停/挂起"（灰色）很长说明有不少工单被搁置。

## 十二、分类规则如何自定义

### 12.1 打开分类规则面板

点击页面右上角的 **"⚙ 分类规则"** 按钮，会从右侧滑出一个配置面板。

### 12.2 面板功能

| 操作 | 说明 |
|------|------|
| **新增分类** | 添加一个新的问题分类（需要填写分类名称和关键词） |
| **编辑分类** | 修改现有分类的名称或关键词（多个关键词用英文逗号或中文逗号分隔） |
| **删除分类** | 移除某个分类（点击 ✕ 按钮） |
| **保存规则** | 保存当前规则配置（需要重新分析才能生效） |
| **保存并重新分析** | 保存规则并立即用新规则重新分析所有工单 |
| **导出规则** | 将当前规则导出为 JSON 文件（可分享给同事） |
| **导入规则** | 从 JSON 文件导入同事分享的规则 |
| **恢复默认** | 重置为系统内置的 7 大分类规则 |

### 12.3 自定义规则示例

假设你想增加一个"数据/计算"的分类来覆盖计算逻辑类问题：

1. 点击"新增分类"

3. 关键词填写：`计算, 数值, 结果, 公式, 算错, 数据`
4. 点击"保存并重新分析"

系统会用新规则重新分析所有工单，原来归入"其他"的工单可能就被正确分类了。

## 十四、数据存储说明

### 14.1 数据存在哪里

上传的工单数据保存在项目目录下的 SQLite 文件中：

```
backend\workorders.sqlite
```

这是一个本地数据库文件，数据不会丢失：
- 页面刷新 → 数据还在
- 后端重启 → 数据还在
- 电脑关机再开机 → 数据还在

### 14.2 如何清空数据

如果需要清空所有数据：
1. 停止后端服务（在终端按 `Ctrl + C`）
2. 删除 `backend\workorders.sqlite` 文件
3. 重新启动后端

### 14.3 上传新数据会怎样

重新上传 Excel **会覆盖当前数据源的所有工单**（按 sourceId 分组删除再插入）。不同数据源之间的数据互不影响。

## 十七、常见问题

### Q1：页面打开一片空白？

**原因**：直接双击打开了 `frontend/index.html`。

**解决**：React + Vite 项目不能直接双击打开 HTML 文件。必须先启动开发服务器：
```bash
npm run dev
```
然后用浏览器访问终端输出的地址（通常是 `http://localhost:5173/`）。

### Q2：页面显示"后端未连接"？

**原因**：后端服务没有启动。

**解决**：
1. 确认后端已启动：`cd backend && npm run dev`
2. 确认看到 `running on http://localhost:3001` 的输出
3. 浏览器打开 `http://localhost:3001/api/health`，应返回 `{ "ok": true }`

### Q3：上传 Excel 提示"找不到工作表"？

**原因**：Excel 文件中没有名为「工单任务」的 Sheet。

**解决**：
- 确认 Sheet 名称完全一致（注意空格、括号等）
- 或者将 Sheet 名称改为包含"工单"二字
- 检查 Excel 文件没有设密码保护

### Q4：上传的 Excel 字段没有被识别？

**原因**：列名与系统期望的不匹配。

**解决**：
- 检查列名是否在[第八章](#八excel-数据格式要求)列出的关键词中
- 可以打开浏览器的开发者工具（F12）→ Console，查看后端日志 `[excelParser] Header mapping`，确认哪些字段被成功映射
- 系统在终端也会输出字段映射结果

### Q5：很多工单被归入"其他"分类？

**原因**：内置关键词不匹配你的实际工单内容。

**解决**：
1. 打开"分类规则"面板
2. 查看当前分类的关键词
3. 根据实际工单内容添加新的关键词或新增分类
4. 点击"保存并重新分析"

### Q6：时间分析区域是空的？

**原因**：Excel 中没有包含时间字段。

**解决**：在 Excel 中增加以下列（任选需要的）：
- 工单提出时间
- 工单解决时间
- 工单验收时间
- 工单归档时间

### Q7：上传新数据后旧数据会怎样？

上传操作会**覆盖**当前数据源的所有数据。如果需要保留历史数据，建议将历史 Excel 文件存档。

### Q8：依赖包在哪里，为什么文件夹这么大？

项目依赖（`node_modules` 文件夹）不会被包含在源码压缩包中。拿到项目后需要在根目录执行：
```bash
npm install
npm run install:all
```
这会下载所有需要的依赖包（约几百 MB）。

### Q9：如何让同事也能访问我的看板？

项目默认只在本机（localhost）运行。如果想让局域网内的同事访问：
1. 需要知道你电脑的局域网 IP（如 `192.168.x.x`）
2. 需要配置防火墙开放对应端口
3. 将前端的 API 地址从 `localhost:3001` 改为你的 IP

> 注意：这是比较进阶的操作，涉及网络安全，建议在有经验的人指导下操作。

### Q10：页面数据突然消失了？

检查 `backend\workorders.sqlite` 文件是否还存在。如果被误删，重新上传 Excel 即可恢复。
